# ローカル生成 `Yuru.app` を `yuru` コマンドの実体として採用する

## 背景

`F22` の出発点は、Yuru を `yuru` コマンドで自然に開けるようにしたい、という要望だった。
単に `electron .` を実行するだけでも起動はできるが、この方法では macOS から見たアプリ名や見た目が
`Electron` 寄りになりやすく、日常的に使う入口としては中途半端だった。
Yuru を「開発中の Electron アプリ」ではなく、「ふだん使う道具」として扱いたいなら、
`Yuru.app` という app bundle を実体にする方向のほうがしっくりくる。

一方で、`Yuru.app` をどう手元に置くかは単純ではなかった。
最初に考えやすいのは、ビルド済みの `.app` を GitHub Release などから配布し、
`yuru` コマンドはそれを開くだけにする構成である。
しかし、今回の検討ではこの方向をいったん外した。
理由は、現時点の Yuru では Apple の公式配布フローに乗るコストが大きいのに対して、
ローカル生成で十分成立する可能性が高かったからである。

今回の議論では、単なる一般論ではなく、実際に app bundle を作って挙動を確認した。
その結果、判断に効いた事実は次の通りだった。

- ローカルで生成した `Yuru.app` は通常起動できた
- 同等の app bundle でも、Chrome 経由で zip をダウンロードして展開すると quarantine が付き、
  Gatekeeper が強くブロックした
- つまり「app bundle にすること」自体が問題なのではなく、
  「未署名 app を配布物としてダウンロードさせること」が UX 上の大きい問題だった

この観察から、当面の Yuru では
「Apple 配布アプリの信頼モデルに寄せる」
よりも
「Node 前提の開発者向けツールとして、利用者のローカルで `Yuru.app` を生成する」
ほうが筋が良いと判断した。

## 決定

Yuru は、管理用 checkout を `~/.yuru/repo` に持ち、
そこから `Yuru.app` をローカル生成して `~/Applications/Yuru.app` に配置する。

コマンドの責務は次のように分ける。

- `yuru`
  `~/Applications/Yuru.app` を開く
- `yuru latest`
  `~/.yuru/repo` を更新し、`npm ci`、build、local packaging を行って
  `~/Applications/Yuru.app` を置き換える
  起動まではしない

インストール経路は `git clone ... && ./install.sh` を前提にし、
`npm link` や `curl | sh` は採らない。

local packaging には `@electron/packager` を使い、
手書きで `Electron.app` の中身を直接組み替える実装は避ける。

## この判断にした理由

### 1. `electron .` のままでは、入口としての見え方が弱い

`electron .` は開発時の起動方法としては自然だが、
日常利用の入口としては `Electron` の存在感が強く残る。
Yuru を「セッション中心のエディタ」として育てていくなら、
起動対象が `Yuru.app` になっているほうが体験として自然だった。

### 2. 未署名 `.app` のダウンロード配布は、現段階の Yuru と相性が悪い

今回の実験で、ローカル生成 app は普通に起動できる一方、
ブラウザ経由でダウンロードした同等の app は Gatekeeper に強く止められることが分かった。
この差はかなり大きく、少なくとも現段階では
「未署名 app を GitHub から落として使ってもらう」
運用は、継続利用の入口として扱いづらい。

問題は app bundle そのものではなく、配布経路と trust model の組み合わせにある。
そのため、app にしたいという要求を満たしつつ、
未署名配布の UX を避ける方法として「ローカル生成」が有力だった。

### 3. Apple の公式配布フローは、今の段階ではコストに見合わない

未署名配布が厳しいなら、正攻法は code signing と notarization である。
しかし、これには Apple Developer Program、証明書管理、配布・更新パイプラインの整備が絡む。

将来的に一般向け配布を本気でやるなら十分ありうるが、
現時点の Yuru はまず自分が日常的に使えることが重要な段階であり、
「正式な macOS プロダクト配布」のための運用コストを先に背負う判断は重すぎる。

今回避けたのは「署名そのもの」ではなく、
「署名・notarization・配布基盤を前提にしないと成立しない更新モデル」である。
Yuru は今のところ、開発者向けローカルツールとして説明したほうが実態に合っている。

### 4. CLI と app の更新元を分けたくなかった

`yuru` コマンドと `Yuru.app` の生成元が別になると、
CLI だけ古い、app だけ新しい、といったずれが起きやすい。
これは更新失敗時の調査や将来の仕様変更時にも地味に効いてくる。

そのため、CLI 側のロジックも local packaging も、同じ管理用 checkout を source of truth とするほうが安定すると判断した。
`~/.yuru/repo` を持つのはこのためである。

### 5. `~/.yuru` と `~/Applications` の役割分担が自然だった

`~/.yuru` には設定や状態も入るが、source は `repo` サブディレクトリに閉じ込めれば役割は整理できる。
一方で実際に使う app は `~/Applications/Yuru.app` に置くことで、
macOS 上でも app として見える場所に置ける。

これにより、
「管理用 checkout は内部領域に置く」
「ふだん開く app は Applications に置く」
という分離ができる。
隠しディレクトリ内の `.app` を直接使うより見通しがよく、
`electron .` のままよりも “道具としての実体” がはっきりする。

## 採らなかった案

### `electron .` をそのまま日常の入口にする

もっとも簡単だが、`Electron` 感が残りやすく、
`yuru` コマンドの実体としては満足度が低かった。

### 未署名の `Yuru.app` をダウンロード配布する

実験結果から、Gatekeeper の扱いが強く、
少なくとも今の段階では継続利用の UX が悪いと判断した。

### Apple の公式配布フローに最初から乗る

将来の選択肢としては残るが、現時点の Yuru に対しては
運用コストと準備コストが重い。
まずは local build ベースの開発者向けツールとして成立させることを優先する。

### `npm link` を主インストール経路にする

仕組みとしては使えるが、ローカル開発中のリンク感が強く、
一時ディレクトリから install した場合の壊れやすさも気になった。
常用の入口としては、専用の install スクリプトで固定配置するほうが自然だった。

### tmp に毎回 clone して build し、app だけ残す

一見きれいだが、毎回 clone するぶん重く、失敗時の調査もしにくい。
CLI と app の生成元も不安定になりやすい。
管理用 checkout を持つほうが、更新経路を理解しやすく保ちやすい。

### npm package と local app を別々に更新する

CLI と app のバージョン感がずれやすいので避けた。
Yuru では、起動ロジックと app 生成ロジックは同じ checkout から来るほうが分かりやすい。

## この判断で受け入れること

- Yuru は当面、Apple 配布アプリではなく「Node 前提の開発者向けローカルツール」として説明する
- local packaging により、署名済み・notarized 配布 app に期待する保証は前提にしない
- `yuru latest` は `main` と依存集合を信頼する運用であり、supply chain の入口は残る
- app をローカル再生成するため、更新ごとに TCC 権限の再許可が必要になる可能性がある

## 今後の課題

この ADR は `F22` の起動・更新方式を決めるものであり、
supply chain をここで解決するものではない。
ただし、懸念が消えたわけではないので、今後の運用課題として明示的に追う。

現時点で候補にしている緩和策は次の通り。

- Renovate の minimum release age によって、出たばかりの依存をすぐ main に入れない
- 依存更新や Yuru 自身の更新に気づけるよう、update notification を入れる
- install / README に trust model を短く明記し、署名済み配布 app と同じではないことを誤認させない

必要になれば、将来ここから
「署名・notarization を前提にした配布モデルへ移るか」
を改めて判断する。
